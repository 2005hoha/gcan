### 子优化0-B：图构建模块

**目标**：实现一个将多智能体状态转换为图结构(G_t)的模块，为GCAN提供输入

**做什么**：
```python
class GraphBuilder:
    """
    输入: [obs_1, obs_2, ..., obs_N]  # N个智能体的局部观测
    输出: Data(x, edge_index, edge_attr)  # PyG的图数据结构
    """
    
    def build(self, observations, agent_positions):
        # 节点特征 x_i = [obs_i, position_i, velocity_i, task_id_i]
        x = self._build_node_features(observations)
        
        # 边: distance_threshold内 → 物理邻近边
        #      comm_range内 → 通信边
        edge_index, edge_attr = self._build_edges(agent_positions)
        
        return Data(x=x, edge_index=edge_index, edge_attr=edge_attr)
```

**关键设计决策**：
- 边类型：物理邻近（距离<d_phys）、通信可能（距离<d_comm）、任务相关（同任务=1否则=0）
- 这些边的阈值d_phys和d_comm是超参数，可以通过域知识设定
- 如果智能体数N较小（<10），可以用全连接图+注意力权重自然稀疏化

**简化的起点方案**（推荐）：
- 先做**全连接图**（所有智能体两两连边）
- 让GAT的注意力机制自动学习哪些边重要（不重要的边注意力权重趋近0）
- 这样可以避免人工设计边类型，同时也是纯数据驱动

**验收标准**：
- 能接受任意N个智能体的观测，输出正确的PyG Data对象
- 图构建时间 < 1ms（不能成为瓶颈）
- 全连接图下，GAT的注意力能在训练后呈现有意义的稀疏化模式

**依赖**：PyG安装完成

**工作量**：2-3天（主要是确定特征维度和验证PyG兼容性）

## 三、Phase 2：GCAN核心实现

### 子优化1-A：GAT信用分配网络（最核心模块）

**目标**：用GAT替代QMIX中简单的单调约束，实现注意力驱动的信用分配

**当前QMIX怎么做**：
```python
# QMIX的混网络：Q_tot = mixing_network(Q_1, Q_2, ..., Q_N | state)
# 约束：∂Q_tot/∂Q_i ≥ 0（每个agent的非负贡献）
# 问题：无法表达智能体间的替代/互补/协调关系
```

**GCAN怎么做**：
```python
class GCAN(nn.Module):
    """GAT驱动的信用分配网络"""
    
    def __init__(self, obs_dim, hidden_dim=128, n_heads=4, n_prototypes=8):
        # GAT编码器
        self.gat1 = GATConv(obs_dim, hidden_dim, heads=n_heads, concat=True)
        self.gat2 = GATConv(hidden_dim*n_heads, hidden_dim, heads=1, concat=False)
        
        # 信用输出头
        self.credit_head = nn.Sequential(
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, 1),
            nn.Softplus()  # 保证信用非负
        )
        
        # 原型相似度（与LLM原型关联）
        self.prototype_proj = nn.Linear(obs_dim, n_prototypes)
    
    def forward(self, graph, agent_q_values):
        # 步骤1: GAT消息传递
        x = F.elu(self.gat1(graph.x, graph.edge_index))
        x = F.elu(self.gat2(x, graph.edge_index))
        
        # 步骤2: 输出每智能体信用分数
        credit_scores = self.credit_head(x)  # [N, 1]
        credit_scores = credit_scores / credit_scores.sum(dim=0)  # 归一化
        
        # 步骤3: 信用加权全局Q值
        Q_tot = (credit_scores.squeeze() * agent_q_values).sum()
        
        return Q_tot, credit_scores, self.gat1.att  # 返回注意力权重用于可视化
```

**设计要点**：

1. **为什么是Softplus而不是ReLU**：Softplus保证信用≥0（满足QMIX的单调约束），但在0附近是平滑的（不截断梯度）

2. **注意力头数(n_heads=4)**：4个头可以并行学习不同类型的信用模式
   - 头1可能学到"空间接近度信用"
   - 头2可能学到"任务完成度信用"
   - 头3可能学到"信息提供度信用"
   - ……

3. **归一化**：credit除以总和，保证数值稳定

4. **返回注意力权重**：直接用于可视化和解释——"agent 3对agent 7的关注度是0.8"意味着agent 3认为agent 7很重要

**训练方式**：
```python
# 与QMIX完全相同的TD损失
td_target = reward + gamma * target_Q_tot(next_graph, target_q_values)
td_error = td_target - Q_tot
loss = td_error ** 2  # MSE
```

**关键**：GCAN不需要额外的损失项。信用分配是隐式学习的——通过最小化TD误差，梯度会自动流向GAT的注意力权重，让"有用的信用分配"得到强化。

**验收标准**：
- 在SimpleSpread(5 agents)上，GCAN的收敛速度≥QMIX
- GCAN的最终性能≥QMIX（理论预测：在5+智能体时超过QMIX）
- 注意力权重可视化有意义（如相近的agent互相关注度高）
- 训练时间不超过QMIX的1.5倍

**工作量**：2-3周（核心开发1周+调试1周+实验验证1周）

### 子优化1-B：LLM信用原型生成

**目标**：让LLM生成K个信用原型，为GCAN的GAT提供语义先验

**设计思路**：

LLM的工作是在训练之前，生成一组"协作角色描述"的文本+对应嵌入向量。

```
LLM输入prompt：
"""
你正在为多智能体协作任务设计信用分配方案。

任务描述：{task_description}
智能体数量：{N}
智能体能力：{capabilities}

请输出K={K}个"信用原型"，每个原型描述一种有价值的协作行为模式。

输出格式（JSON）：
[
  {
    "name": "领航探索者",
    "description": "率先接近目标区域，为团队提供前方信息的智能体",
    "behavioral_indicators": ["移动速度快", "位置在团队前方", "视野覆盖未知区域"]
  },
  ...
]
"""

获取原型后：
1. 将每个原型的description用Sentence-BERT编码为向量p_k ∈ R^d
2. 这K个向量p_1, ..., p_K固定，作为GCAN中prototype_proj的目标
3. 训练时，GCAN的prototype_proj学习将智能体特征映射到原型空间
   - 即：sim(agent_i, prototype_k) = cos(prototype_proj(obs_i), p_k)
   - 智能体与哪个原型最相似，就说明它当前主要扮演哪种角色
```

**为什么需要LLM**：如果不用LLM，K个原型是随机的，没有语义。LLM用自然语言理解任务后，能生成"有意义的"角色分类。这是LLM在该系统中唯一的语义贡献点——一次性的、低成本的。

**简化的起步方案**（如果LLM API不可用）：
- 手动定义K个原型（根据对任务的理解）
- 例如在SimpleSpread中："最接近目标1的智能体"、"最接近目标2的智能体"、……
- 先用人工原型跑通GCAN全流程，确认架构正确后再接入LLM

**验收标准**：
- LLM生成的K个原型语义互不相同（检查cosine相似度）
- 人工评估原型的合理性
- LLM调用仅需1次（训练开始时），成本 << 单次训练的费用

**工作量**：2-3天（prompt工程+验证，LLM部分可以后做）

### 子优化1-C：GCAN+QMIX联合对比实验

**目标**：在同一环境下公平对比GCAN vs QMIX vs QMIX+LCA

**对比维度**：

| 维度 | 为什么重要 | 如何测量 |
|------|----------|---------|
| 收敛速度 | GCAN应该更快（更好的信用信号→更有效的梯度） | 达到阈值性能的训练步数 |
| 最终性能 | GCAN应该≥QMIX | 最后10%训练步的平均回报 |
| 扩展性 | GCAN在N增大时优势更明显 | 固定环境，变化N=3,5,8,12 |
| 解释性 | GAT注意力权重应可视化为有意义的模式 | 对比信用分配与人工判断的一致性 |

**实验配置**：
```
环境：SimpleSpread（EPyMARL内置）
智能体数：3, 5, 8, 12
训练步数：2M per config
随机种子：3个
对比算法：QMIX(基线), GCAN(我们的), GCAN-no-proto(消融:无LLM原型)
```

**预测**：
- N=3：GCAN ≈ QMIX（简单任务不需要复杂信用分配）
- N=5：GCAN > QMIX（角色开始分化，GAT捕捉到交互模式）
- N=8：GCAN >> QMIX（QMIX的单调分解在多智能体下严重受限）
- N=12：GCAN仍能工作，QMIX可能不收敛

**验收标准**：
- 至少有一个N值下GCAN显著优于QMIX（p<0.05，t检验）
- 消融实验说明LLM原型的贡献（GCAN vs GCAN-no-proto）
- 生成注意力权重热力图（可放入论文）

**工作量**：1-2周（主要是跑实验和调参）

## 四、Phase 3：LLM智能调度

### 子优化2-A：HTA-Lite事件驱动LLM

**目标**：替代固定周期调用LLM，用简单事件触发器

**做什么**：

```python
class HTALiteScheduler:
    """
    事件驱动的LLM调用调度器。
    基于规则（不需要训练），作为RALLY固定50步的简单替代。
    """
    
    def __init__(self, 
                 deviation_threshold=5.0,    # 任务偏离阈值(米)
                 safety_threshold=3.0,       # 安全临界阈值(米)
                 max_steps_without_llm=200):  # 兜底最大间隔
        self.thresholds = {...}
        self.last_llm_call = 0
        self.llm_plan = None
    
    def should_call_llm(self, step, agents_state):
        """判断当前是否应该调用LLM"""
        
        # 事件1: 任务偏离
        if self.llm_plan is not None:
            deviation = self._compute_deviation(agents_state, self.llm_plan)
            if deviation > self.deviation_threshold:
                return True, "deviation"
        
        # 事件2: 安全临界
        min_dist = self._min_agent_distance(agents_state)
        if min_dist < self.safety_threshold:
            return True, "safety_critical"
        
        # 事件3: 超时兜底
        if step - self.last_llm_call > self.max_steps_without_llm:
            return True, "timeout"
        
        return False, None
    
    def _compute_deviation(self, state, plan):
        """计算实际位置与LLM规划的偏差"""
        actual_pos = state["positions"]      # [N, 2/3]
        planned_pos = plan["target_positions"]  # [N, 2/3]
        return torch.norm(actual_pos - planned_pos, dim=-1).mean()
```

**预期效果**：
- 在平稳巡航阶段：LLM调用间隔从50步增加到200+步（节省~75% LLM调用）
- 在危险/偏离阶段：LLM可以随时被触发（不受50步窗口限制）
- 在RALLY的搜索任务中：遇到障碍时立即重规划，无需等待50步窗口

**验收标准**：
- LLM调用次数减少 >50%（与RALLY固定50步对比，同等任务性能下）
- 安全事件发生时LLM响应延迟 <1步（立即触发）
- 任务性能不下降（与固定50步baseline对比）

**工作量**：1周（规则逻辑简单，主要是调阈值）

### 子优化2-B：GateNet门控学习

**目标**：用学习替代规则，训练一个轻量GNN预测"现在需要LLM吗"

**设计思路**：在HTA-Lite的规则基础上增加学习组件。

```python
class GateNet(nn.Module):
    """轻量GNN门控网络"""
    
    def __init__(self, obs_dim, hidden_dim=64):
        # 小GAT（比GCAN的GAT小，因为任务更简单）
        self.gat = GATConv(obs_dim, hidden_dim, heads=2)
        self.gate_head = nn.Sequential(
            nn.Linear(hidden_dim*2, 32),
            nn.ReLU(),
            nn.Linear(32, 1),
            nn.Sigmoid()
        )
    
    def forward(self, graph):
        x = F.elu(self.gat(graph.x, graph.edge_index))
        gate = self.gate_head(x)  # [N, 1]，每个智能体的门控值
        return gate
```

**训练方式**（离线，不干扰主训练循环）：

```
数据收集阶段：
  运行多次训练，每次用不同的LLM调用频率
  记录：(state, 是否调用了LLM, 调用后K步内的性能变化ΔR)
  
  标签构造：
    如果ΔR > δ（调用LLM显著改善了性能）→ label=1（应该调用）
    如果ΔR ≈ 0（调用LLM无影响）→ label=0（不用调用）
    如果ΔR < 0（调用LLM反而更差）→ label=0（不用调用）

监督学习：
  loss = BCE(gate_prediction, label)
```

**关键设计**：GateNet不参与RL训练。它是独立训练的，只在部署时使用。这样不会干扰主策略的优化。

**与HTA-Lite的集成**：
```python
# 综合决策：规则+学习
def should_call_llm_hybrid(step, state, graph):
    rule_result, reason = hta_lite.should_call_llm(step, state)
    
    if rule_result:
        return True, f"rule:{reason}"  # 规则触发→立即调用
    
    gate_value = gatenet(graph).mean()
    if gate_value > 0.7:
        return True, f"learned:{gate_value:.2f}"  # 学习触发→调用
    
    return False, None
```

**验收标准**：
- GateNet的预测准确率 >70%（在保留测试数据上）
- GateNet+HTA-Lite的LLM调用效率 > 纯HTA-Lite
- GateNet的推理时间 <5ms（不能成为瓶颈）

**工作量**：1-2周

### 子优化2-C：LLM决策蒸馏（可选）

## 五、Phase 4：信号健康度

### 子优化3-A：CoEvo-Lite信号监测

**目标**：在训练过程中检测LLM信号（奖励函数/子任务分配）是否仍然有效

**做什么**：

```python
class SignalHealthMonitor:
    """
    监测LLM生成信号的质量和时效性
    """
    
    def __init__(self, window_size=20, saturation_threshold=0.1):
        self.reward_history = []
        self.entropy_history = []
        self.window_size = window_size
    
    def update(self, rewards, policy_entropy):
        self.reward_history.append(rewards.mean())
        self.entropy_history.append(policy_entropy)
        
        if len(self.reward_history) > 100:
            self.reward_history.pop(0)
            self.entropy_history.pop(0)
    
    def check_health(self):
        """返回信号健康状态"""
        
        if len(self.reward_history) < self.window_size:
            return "healthy", 1.0  # 数据不足，假设健康
        
        # 指标1：奖励方差趋势
        recent_var = np.var(self.reward_history[-self.window_size:])
        early_var = np.var(self.reward_history[-2*self.window_size:-self.window_size])
        var_ratio = recent_var / (early_var + 1e-8)
        
        # 指标2：策略熵趋势
        recent_entropy = np.mean(self.entropy_history[-self.window_size:])
        
        # 综合判断
        if var_ratio < self.saturation_threshold:  # 方差剧烈下降=饱和
            return "saturated", var_ratio
        elif recent_entropy < 0.01:  # 策略几乎确定性=不再探索
            return "stale", recent_entropy
        else:
            return "healthy", max(var_ratio, recent_entropy)
    
    def should_regenerate_signal(self):
        status, score = self.check_health()
        return status in ["saturated", "stale"]
```

**触发修正的策略**：
```python
if monitor.should_regenerate_signal():
    # 收集代表性轨迹
    trajectories = collect_representative_trajectories(policy, env, n=10)
    
    # 发送给LLM
    new_signal = llm.regenerate_signal(
        task_description=task_desc,
        current_policy_performance=current_metrics,
        failure_cases=trajectories.failure_cases,
        success_cases=trajectories.success_cases
    )
    
    # 在小规模验证环境上对比新旧信号
    if validate_signal(new_signal, old_signal, env_subset):
        update_signal(new_signal)
        monitor.reset()
```

**验收标准**：
- 在饱和事件发生后5轮内检测到
- 误报率 <20%（不应频繁打断正常的训练）
- 信号更新后性能有明显恢复（>5%提升）

**工作量**：1-2周
